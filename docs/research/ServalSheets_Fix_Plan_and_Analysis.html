2. Range Format Inconsistency — The #1 LLM Usability Issue

The range field uses 3 different type systems across 25 tools — the single biggest source of first-call failures for LLM callers.

PatternTypeTools Using ItActions AffectedLLM Impact
RangeInputSchema Union: string auto-wrapped to {a1:""} + named + semantic + grid sheets_data, sheets_format, sheets_dimensions, sheets_advanced, sheets_analyze, sheets_bigquery ~70 actions Flexible — but z.preprocess may not fire consistently
A1NotationSchema Plain string with regex validation sheets_compute (10), sheets_quality (2) 12 actions Strict string-only — rejects {a1:""} objects
z.string() Plain string, minimal validation sheets_fix (5), sheets_history (2), sheets_composite (3), sheets_templates (1), sheets_connectors (1) 12 actions Accepts anything — but no A1 validation, so garbage in/garbage out

Recommended Fix: Universal Range Normalization

// In src/schemas/shared.ts — create a SINGLE UniversalRangeSchema
export const UniversalRangeSchema = z.preprocess(
  (val) => {
    if (typeof val === 'string') return { a1: val };  // Always wrap strings
    return val;
  },
  z.union([
    z.object({ a1: A1NotationSchema }),
    z.object({ namedRange: z.string().min(1) }),
    z.object({ semantic: SemanticRangeQuerySchema }),
    z.object({ grid: GridRangeSchema }),
  ])
);

// Then: replace ALL z.string() ranges AND A1NotationSchema ranges
// with UniversalRangeSchema across these files:
// - src/schemas/compute.ts (10 actions)
// - src/schemas/quality.ts (2 actions)
// - src/schemas/fix.ts (5 actions)
// - src/schemas/history.ts (2 actions)
// - src/schemas/composite.ts (3 actions)

// Handler-side: add extractA1() helper that unwraps any format to string
export function extractA1(range: RangeInput): string {
  if ('a1' in range) return range.a1;
  if ('namedRange' in range) return range.namedRange;
  if ('grid' in range) return gridToA1(range.grid);
  if ('semantic' in range) throw new ValidationError('Semantic ranges not supported for this action');
}

Impact: ~24 actions become consistently callable. LLMs can always send plain strings or objects. Files changed: 6 schema files + 5 handler files. Run npm run schema:commit after.

5. Intelligent Routing & Usability Improvements

5.1 — Auto-Route to Tasks API Based on Estimated Complexity

The server already has Tasks API support (SEP-1686) for 9 tools. Add automatic complexity estimation to route heavy operations to background execution:

// NEW: src/middleware/auto-task-routing.ts
export function shouldRouteToTask(toolName: string, action: string, input: any): boolean {
  // Rule 1: BigQuery import/export always → task
  if (toolName === 'sheets_bigquery' && ['export_to_bigquery', 'import_from_bigquery'].includes(action)) return true;

  // Rule 2: Full-sheet scans on large sheets → task
  if (['analyze_formulas', 'list_chips', 'analyze_performance'].includes(action)) {
    const metadata = getCachedMetadata(input.spreadsheetId);
    if (metadata?.totalCells > 10000) return true;
  }

  // Rule 3: Timeline with many revisions → task
  if (action === 'timeline') return true;

  // Rule 4: Composite ops that estimate > 30s
  if (toolName === 'sheets_composite' && ['setup_sheet', 'export_large_dataset'].includes(action)) {
    const metadata = getCachedMetadata(input.spreadsheetId);
    if (metadata?.totalRows > 10000) return true;
  }

  return false;
}
